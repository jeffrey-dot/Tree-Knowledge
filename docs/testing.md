# 测试规范

## 目标
这份文档用于定义 Tree Knowledge 的测试策略、测试层级和最低执行标准。  
目标不是追求测试数量，而是确保核心知识流、结构编辑、Provider 配置和本地数据读写在持续演进中不被破坏。

## 总原则
- 所有实现性变更都必须考虑测试
- 优先测试核心行为，不优先测试实现细节
- 能写自动化测试的，不用手工验证替代
- 无法补自动化测试时，必须明确说明原因和风险
- 修复 bug 时，优先补回归测试

## 测试分层
MVP 采用四层测试结构：

### 1. 单元测试
目标：
- 验证纯逻辑函数、数据转换、校验器和 Provider 适配层

适用范围：
- `lib/validators`
- `lib/llm`
- `lib/providers`
- 图谱布局辅助逻辑
- 节点落位判断逻辑

要求：
- 纯函数优先覆盖
- 所有关键枚举分支都要测
- 所有 Zod schema 至少要有一组成功和一组失败用例

### 2. 集成测试
目标：
- 验证主进程服务、SQLite 数据层、Drizzle 查询和应用命令的协作结果

适用范围：
- `createWorkspace`
- `createRootNode`
- `generateCandidates`
- `expandNode`
- `moveNode`
- `createEdge`
- `getWorkspaceSnapshot`

要求：
- 使用测试数据库或临时 SQLite 文件
- 每条命令至少有一条成功用例
- 核心失败路径必须覆盖

### 3. 组件测试
目标：
- 验证桌面渲染层中的关键视图组件是否按状态正确渲染和响应交互

适用范围：
- 启动台卡片流
- Provider 管理表单
- 工作台左栏导航
- 节点详情面板
- 候选节点确认层

要求：
- 测试重点是状态变化和关键交互
- 不测试视觉实现细枝末节
- 不做大量快照堆积

### 4. 端到端测试
目标：
- 验证用户从桌面启动台到知识工作台的核心主链路

适用范围：
- 创建知识库
- 配置 Provider
- 生成根节点
- 继续扩展节点
- 回跳节点
- 搜索节点

要求：
- 只覆盖核心主路径，不追求全量覆盖
- 每个核心功能至少有一条端到端主流程

## 推荐工具
### 单元 / 集成 / 组件测试
- `Vitest`
- `@testing-library/react`
- `@testing-library/user-event`

### 端到端测试
- `Playwright`

### 数据层测试
- 临时 `SQLite` 数据文件
- 或每次测试动态创建隔离数据库

## 必测范围
以下能力属于必须覆盖的 MVP 核心：

### 知识库与节点
- 创建知识库后能正确创建根节点
- 根节点不会错误写入层级父关系
- 节点扩展时能正确创建 `child / branch / related`
- 节点移动父级后路径和快照同步更新

### 工作台聚合读取
- `workspace-snapshot` 能正确返回当前节点、祖先、子节点、相关节点
- 归档节点不会出现在默认工作台结果中

### 候选节点流程
- 生成候选不会污染正式节点表
- 采纳候选后才会创建正式节点
- 非法候选 ID 不能被错误采纳

### Provider 配置
- Provider 能保存、读取、更新
- 缺失关键字段时校验失败
- Provider 连通性测试失败时能返回明确错误
- 默认模型映射能正确保存

### LLM 输出校验
- 根节点生成输出结构校验
- 候选节点输出结构校验
- 直接生成节点输出结构校验
- 快照输出结构校验

### 搜索
- 标题搜索命中
- 摘要搜索命中
- 搜索结果带路径信息

## Bug 修复规则
- 每个可复现 bug 修复都应优先补一个自动化回归测试
- 如果 bug 位于 UI 但根因在服务层，优先在服务层补测试
- 如果 bug 只能在端到端场景中稳定复现，就补 E2E

## 不建议的测试方式
- 大量无断言的“跑一遍”测试
- 大量脆弱快照测试
- 只测 happy path，不测失败路径
- 把手工点击记录当成正式测试替代品

## 最低提交标准
任何实现性 PR 或提交至少要满足以下之一：

1. 新增或修改了自动化测试
2. 明确说明为什么当前无法补自动化测试
3. 写清手工验证步骤和剩余风险

如果改动涉及以下模块，则默认必须补自动化测试：
- 数据模型
- 主进程命令/查询
- Provider 配置
- LLM 输出解析
- 工作台核心状态逻辑

## AGENTS 执行要求
对参与实现的 Agent，测试要求如下：

- 实现新功能时，必须同步评估需要新增哪些测试
- 修改已有逻辑时，必须检查现有测试是否需要更新
- 提交实现结果时，必须说明：
  - 新增了哪些测试
  - 运行了哪些测试
  - 哪些测试没跑
  - 剩余风险是什么
- 如果没有补测试，必须明确说明原因，不能省略

## 建议目录
建议采用以下测试目录结构：

```text
tests/
  unit/
  integration/
  component/
  e2e/
```

## MVP 建议优先级
### P0
- Provider 配置集成测试
- 节点创建 / 扩展 / 移动集成测试
- 工作台聚合读取集成测试
- LLM 输出 schema 单元测试

### P1
- 启动台组件测试
- 工作台关键组件测试
- 端到端主链路测试

### P2
- 更细的图谱交互测试
- 更多异常状态 E2E
