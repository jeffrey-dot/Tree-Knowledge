# Tree Knowledge 愿景

## 背景问题
现有的 LLM 问答界面默认采用线性聊天流。这个形式适合连续对话，但不适合知识沉淀，主要问题有：

- 同一主题在多轮追问后很难快速回到某个关键节点
- 中途切换方向时，新问题会继续堆叠在原有聊天流中
- 用户想沿着旧问题继续深挖时，需要反复向上翻历史消息
- 多个相关主题之间缺少明确的结构连接
- 模型依赖整段聊天作为上下文，随着长度增长，质量和可控性都会下降

这意味着用户虽然在不断提问，但产出物仍然是难以复用的消息记录，而不是可导航、可维护的知识结构。

## 产品目标
Tree Knowledge 要把“问答过程”转化为“知识节点网络”。用户可以从一个问题出发，沿着任意节点不断展开，形成既有主路径又有横向连接的知识图谱。

MVP 的核心目标：
- 让用户始终知道当前知识位置
- 让用户可以从任意历史节点重新出发
- 让知识扩展成分支，而不是退化成长聊天
- 让节点摘要成为后续推理和生成的稳定上下文

## 非目标
以下内容不属于 MVP 目标：
- 替代通用聊天产品
- 构建完整团队知识管理平台
- 一开始就支持海量外部内容采集
- 做成复杂图数据库分析产品
- 让 LLM 自动无限生长而无需用户确认

## 核心概念
### 节点
节点是系统中的主资产。每个节点代表一个可独立命名和沉淀的知识单元，通常包含：
- 标题
- 摘要
- 正文
- 主父节点
- 关系边
- 生成来源

### 层级
层级用于表达“这个问题/主题是从哪里继续展开的”。  
每个节点只能有一个主父节点，这样才能保证主路径清晰、回跳稳定。

### 关系边
关系边用于表达节点之间的横向连接，例如支持、对比、依赖、例子。  
它不参与主路径定位，但用于扩展联想和图谱可视化。

### 当前节点
当前节点是工作台的中心对象。图谱画布、详情区、输入区和 LLM 上下文都以它为中心组织。

### 候选节点
候选节点是 LLM 基于当前节点与用户意图提出的扩展建议。  
它们不是正式知识资产，只有用户确认后才会创建为正式节点。

## 产品主逻辑
系统的默认工作流如下：

1. 用户输入一个问题
2. 系统创建根节点
3. 用户围绕当前节点继续提问
4. 系统可直接生成新节点，或先给出多个候选扩展方向
5. 用户确认后，节点作为 `子节点`、`平行分支` 或 `关联节点` 落入图谱
6. 用户可以随时跳回任意祖先节点重新展开
7. 用户可以用关系边把不同分支重新连接起来

这个逻辑的关键不是“记录每次问了什么”，而是“每次问完后图谱怎么长”。

## 设计原则
### 节点是资产，消息是材料
系统主导航对象必须是节点，而不是聊天消息。消息只用于解释节点如何产生，不能承担主要结构职责。

### 树负责定位，边负责联想
树结构解决“我从哪来、我现在在哪”，关系边解决“我还可以关联到哪里”。

### 用户确认结构，LLM 提供建议
LLM 可以建议节点标题、摘要、归类和下一步扩展方向，但不能代替用户决定全局结构。

### 当前节点优先于全局上下文
每次生成时都应围绕当前节点及其祖先摘要组织上下文，不应直接传入整段长对话。

### 图谱是主视图，不代表无限复杂
产品主入口应是图谱工作台，但 MVP 只展示有限半径节点，避免可视化先失控。

## 成功标准
MVP 应满足以下结果：
- 用户从一个根问题扩展 10 到 50 个节点后仍不混乱
- 用户能在 2 次点击内回到任意祖先节点
- 同一主题的多轮追问会沉淀成节点，而不是长消息串
- LLM 生成出的新节点能够稳定挂接到当前结构中

## 失败信号
如果出现以下现象，说明产品正在退化回线性聊天：
- 工作台核心区域实际变成消息列表
- 新问题默认追加为聊天消息而非节点
- 用户回跳依赖滚动聊天历史而不是切换节点
- LLM 上下文严重依赖整段消息历史
- 图谱只是装饰，无法承载真实导航行为
